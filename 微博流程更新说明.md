# 微博流程更新说明

## 一、更新内容

根据需求，在选中两张图片后，增加了第二次"下一步"点击功能。

## 二、完整流程

```
1. 点击"首页"
   ↓
2. 点击微博加号入口
   ↓
3. 点击"图片"
   ↓
4. 等待图片加载（1500ms）
   ↓
5. 选中第二张和第三张图片
   ↓
6. 延迟500ms
   ↓
7. 【第一次点击"下一步"】← 从相册选择页面进入图片编辑页面
   ↓
8. 等待新页面加载（2000ms）← 新增延时
   ↓
9. 【第二次点击"下一步"】← 从图片编辑页面进入发布页面 ← 新增步骤
   ↓
10. 流程完成
```

## 四、关键点说明

### 1. 第一次点击"下一步"
- **位置**: 相册选择页面
- **功能**: 从相册选择页面进入图片编辑页面
- **延时**: 选中图片后延迟500ms

### 2. 页面加载延时
- **延时时间**: 2000ms（2秒）
- **目的**: 等待新页面（图片编辑页面）完全加载完成
- **重要性**: 确保第二次点击"下一步"时，页面已经准备好

### 3. 第二次点击"下一步"
- **位置**: 图片编辑页面
- **功能**: 从图片编辑页面进入发布页面
- **错误处理**: 如果找不到按钮，记录日志并退出流程

## 五、页面流转

```
相册选择页面
   ↓ [点击"下一步"]
图片编辑页面（等待2秒）
   ↓ [点击"下一步"]
发布页面
```

## 六、日志输出

流程中会输出以下日志信息：

1. `"已点击第一步的下一步，等待页面加载"` - 第一次点击后
2. `"准备点击第二步的下一步"` - 准备第二次点击前
3. `"已点击第二步的下一步，流程完成"` - 第二次点击成功
4. `"未找到第二步的\"下一步\"按钮"` - 如果第二次点击失败

## 七、延时调整建议

如果发现延时不够或太长，可以调整 `delay(2000)` 中的数值：

- **延时太短**：页面未加载完成，第二次点击可能失败
  - 建议增加延时：`delay(3000)` 或 `delay(2500)`

- **延时太长**：流程执行较慢
  - 建议减少延时：`delay(1500)` 或 `delay(1800)`

可以根据实际测试情况调整这个值。

## 八、错误处理

### 第一次点击失败
- 日志：`"未找到\"下一步\"按钮"`
- 行为：直接退出流程

### 第二次点击失败
- 日志：`"未找到第二步的\"下一步\"按钮"`
- 行为：直接退出流程

## 十、测试建议

1. **测试页面加载时间**：确认2秒延时是否足够
2. **检查按钮文本**：确认两个页面的"下一步"按钮文本是否一致
3. **验证流程**：完整运行一次，确认两个"下一步"都能成功点击
4. **日志监控**：查看日志输出，确认每一步都正常执行

## 十一、可能的优化

### 1. 动态等待页面加载
可以检查页面特定元素来判断页面是否加载完成，而不是固定延时：

```kotlin
// 等待页面加载（检查特定元素是否存在）
var pageLoaded = false
var waitCount = 0
while (!pageLoaded && waitCount < 20) {  // 最多等待4秒（20*200ms）
    delay(200)
    waitCount++
    // 检查图片编辑页面的特征元素
    if (AssistsCore.findByText("编辑").isNotEmpty() || 
        AssistsCore.findByText("完成").isNotEmpty()) {
        pageLoaded = true
    }
}
```
